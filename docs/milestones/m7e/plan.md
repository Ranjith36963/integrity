## Milestone 7e — First-brick text card + brand-mark long-press easter egg + toasts — Plan

### Context

M7e is the **final M7 chunk** and closes the M7 motion-polish epic. It bundles three tiny narrative surfaces — each independent, each ships together because none earns its own milestone:

1. A one-time **"Your Empire begins." text card** that fires on the user's very first brick add (across all program history, persisted via a single boolean so it never re-fires).
2. A 600 ms **long-press on the top-bar brand mark** ("DHARMA" wordmark in `<TopBar>`) that reveals a year-heatmap preview overlay (re-uses M9e's `<YearView>` visual vocabulary at smaller scale); release closes it.
3. A global **`<Toaster>`** that fires subtle 2 s bottom-anchored toasts on `ADD_BLOCK` / `DELETE_BLOCK_*` / `ADD_BRICK` / `DELETE_BRICK`; single-toast-at-a-time, last-write-wins.

M7e is **render-layer + a single persisted boolean.** A new optional `firstBrickShown?: boolean` is added to `AppState` and `PersistedState`; this is an **additive lossless v3-payload extension** (per ADR-044's "optional fields with defaults are an additive change" pattern), so `SCHEMA_VERSION` stays at `3` — no migrator step, no new ADR. The reducer's `ADD_BRICK` arm gains an idempotent flip (first hit only; further dispatches no-op on the flag). Existing saved states are handled by an additive read-time projection in `lib/persist.ts`'s v3 case: if `firstBrickShown` is absent on a v3 payload, it is back-filled to **`true` if any brick exists in `state.blocks[].bricks` or `state.looseBricks`, otherwise `false`** — so existing users with bricks never see the card retroactively (spec AC #5).

### Feature grouping

m7e is **one feature group** (slug `m7e`), one BUILDER dispatch (ADR-013, ADR-022). `tests.md` IDs use the `m7e` slug (`U-m7e-*`, `C-m7e-*`, `E-m7e-*`, `A-m7e-*`). See § Test-ID prefix scheme.

### Data model

**One additive optional field added to both `AppState` (`lib/types.ts`) and `PersistedState` (`lib/persist.ts`):**

```ts
// lib/types.ts — additive
export type AppState = {
  blocks: Block[];
  categories: Category[];
  looseBricks: Brick[];
  programStart: string;
  currentDate: string;
  history: Record<string, ArchivedDay>;
  deletions: Record<string, true>;
  firstBrickShown?: boolean; // M7e — one-time narrative flag (ADR-039); flipped by ADD_BRICK arm.
};

// lib/persist.ts — additive (no SCHEMA_VERSION bump)
export type PersistedState = {
  schemaVersion: 3;
  programStart: string;
  currentDate: string;
  history: Record<string, ArchivedDay>;
  blocks: Block[];
  categories: Category[];
  looseBricks: Brick[];
  deletions: Record<string, true>;
  firstBrickShown?: boolean; // M7e — additive within v3 (ADR-044 "optional fields are an additive change").
};
```

**No `SCHEMA_VERSION` bump (stays at `3`).** Per ADR-044, "an optional field with a defaulted read is an additive lossless extension within version." M9b's v1→v2 and M5's v2→v3 each added **required** fields/maps (`history`, `deletions`); M7e adds **one optional boolean** with a deterministic back-fill rule (`hasAnyBrick(blocks, looseBricks)`). The read-time projection in `migrate(raw)`'s v3 case handles existing v3 payloads that pre-date M7e (back-fills to `true` if bricks exist, `false` if not; spec AC #5).

**No new action.** The reducer's existing `ADD_BRICK` arm flips the flag inline.

**Persistence migrator (additive read-time projection in `lib/persist.ts`):**

```ts
// inside migrate(raw)'s case 3 (v3 terminus) — sketch
const hasAnyBrick =
  blocks.some((b) => b.bricks.length > 0) || looseBricks.length > 0;
const firstBrickShown =
  typeof obj.firstBrickShown === "boolean" ? obj.firstBrickShown : hasAnyBrick; // back-fill: existing users with bricks see no card
return {
  /* ... */ firstBrickShown,
};
```

The same back-fill rule is applied in case 2 (v2 → v3) — existing v2 payloads have bricks AND have no `firstBrickShown` → projection back-fills to `true`. Case 1 (v1 → v3) has no `looseBricks` (v1 predates them) and no inside-block bricks (M2's `ADD_BLOCK` arm didn't populate `bricks` either) — projection lands at `false`, which is correct for the only realistic v1 holder (a developer running the very first build).

**`defaultPersisted()` returns `firstBrickShown: false`** — first-run users have no bricks; the card fires on their genuine first `ADD_BRICK`.

**`<Toaster>` (NEW — `components/Toaster.tsx`)**

- **Singleton mount.** `<Toaster />` mounts inside `<AppShell>`. Module-level event emitter:
  ```ts
  type ToastKind = "success" | "info" | "error";
  type ToastEvent = { id: string; message: string; kind: ToastKind };
  const subscribers = new Set<(t: ToastEvent | null) => void>();
  export function toast(message: string, kind: ToastKind = "info"): void {
    const event: ToastEvent = { id: crypto.randomUUID(), message, kind };
    subscribers.forEach((fn) => fn(event));
  }
  ```
- **Subscribe pattern:** `<Toaster />` `useEffect` adds an `(event) => setCurrent(event)` callback to the Set on mount; removes on unmount. The callback resets the auto-dismiss timer on every emit (last-write-wins).
- **DOM (when `current !== null`):** a Framer `<motion.div>` with `data-testid="toast"`. `role="status"` + `aria-live="polite"` for success/info; `role="alert"` + `aria-live="assertive"` for error (resolves SG-m7e-03).

- **Color by kind:** success → 1 px border `var(--accent)`; info → 1 px border `var(--ink-dim)`; error → 1 px border `var(--cat-9)` (the red token in M0's category palette, WCAG-AA on `var(--card)`).
- **Single toast at a time.** Every `toast(...)` call replaces the current one (no queue). The auto-dismiss timer (2000 ms) is reset on every emit.

### Toast wiring

The four toast events fire AFTER the corresponding `dispatch(...)` call inside `<BuildingClient>`:

| Action        | Call site in BuildingClient.tsx                                 | toast(...) call                     |
| ------------- | --------------------------------------------------------------- | ----------------------------------- |
| Block created | `handleChooserPick` "block" branch                              | `toast("Block created", "success")` |
| Block deleted | `handleConfirmJustToday` / `handleConfirmAll` (M5 confirm flow) | `toast("Block deleted", "info")`    |
| Brick added   | `handleChooserPick` "brick" branch                              | `toast("Brick added", "success")`   |
| Brick deleted | `handleConfirmDelete` (M5 brick path)                           | `toast("Brick deleted", "info")`    |

**Rolled-back action (spec AC #11) → error toast.** Today's reducer (lines 175–193 + 195–217) silently returns `state` unchanged when a delete targets a missing id; the dispatch site has no way to detect that. **For M7e, we treat dispatch-then-toast as fire-and-forget on the happy path** — the four toast call sites above run unconditionally. The spec AC #11 "rolled-back action emits an error toast" is genuinely satisfied today only by the `REORDER_BLOCK` overlap-rejection path (M6) which already has a haptic + aria-live announce; M7e does NOT wire toast emit into REORDER_BLOCK because no spec'd event in M7e covers reorder. **PLANNER flags this as SG-m7e-05** for VERIFIER (see § Open questions for VERIFIER) — if the user wants the M6 overlap-rejection path to fire an error toast too, that's a one-line addition at the `DraggableTimelineBlock` snap-back call site.

### Resolutions to the 4 Open Spec Gaps

- **SG-m7e-01 — RESOLVED → top-level `AppState.firstBrickShown?: boolean`, additive migrator, no schema bump.** Added to both `AppState` and `PersistedState` as optional. The `migrate(raw)` v3 case back-fills via `hasAnyBrick(blocks, looseBricks)` for payloads that pre-date M7e; first-run defaults via `defaultPersisted()` to `false`. `SCHEMA_VERSION` stays at `3` per ADR-044's "optional fields are an additive change" clause. VERIFIER: confirm `lib/persist.test.ts` covers all four payload shapes (absent + has-bricks, absent + no-bricks, `true`, `false`) and that no migrator-bump test is added.

- **SG-m7e-02 — RESOLVED → compose `<MonthCell>` directly in `<YearHeatmapPreview>`; do NOT extend `<YearView>` with a `size` prop.** Rationale: extending the shipped M9e `<YearView>` with a `size="full" \| "preview"` prop touches a stable surface and adds a new render path (every existing M9e test would need to assert the default `size="full"`). Composing the existing `<MonthCell>` primitive directly is a smaller diff, has zero impact on M9e's surface, and keeps the preview overlay's styling local to one file. VERIFIER may pick the prop-extension alternative if duplication looks excessive; same observable behavior. **No `<YearView>` modification in M7e.**

- **SG-m7e-03 — RESOLVED → `role="status"` + `aria-live="polite"` for success/info; `role="alert"` + `aria-live="assertive"` for error.** Wired in `<Toaster>` based on the `kind` discriminant. VERIFIER: run axe on each kind variant; confirm screen-reader announcement timing in `<Toaster.test.tsx>`.

- **SG-m7e-04 — RESOLVED → toast at z-30; first-brick card at z-40; year-heatmap overlay at z-40. Dock is z-20; sheets/modals are z-50.** Stacking order from top to bottom: sheets/modals (z-50) > first-brick card + year-heatmap preview (z-40) > toast (z-30) > dock (z-20). The bottom-anchor offset `bottom: calc(80px + env(safe-area-inset-bottom))` keeps toasts visually above the dock without overlapping it. VERIFIER: confirm visual stacking on iPhone 12 (430 px) viewport during Gate #2 tap-test.

### Accessibility

- **`<FirstBrickCard>`:** `role="status"` + `aria-live="polite"` — single polite announcement of "Your Empire begins." Tap-target is the full card body (≥ 44 px ADR-031). No focus trap (decorative, auto-dismiss).
- **`<BrandMarkLongPress>`:** brand-mark wrapper becomes a `<button>` element with `aria-label="DHARMA — long-press for year heatmap"`. Tappable area ≥ 44 px tall via padding (ADR-031). Touch-driven; pointer-events.
- **`<YearHeatmapPreview>`:** `role="dialog"` + `aria-modal="true"` + `aria-label="Year heatmap preview"`. Grid is `aria-hidden="true"` (peek-only — release closes). No focus trap (no interactive element inside).
- **`<Toaster>`:** `role="status"` + `aria-live="polite"` for success/info; `role="alert"` + `aria-live="assertive"` for error. AT users hear the toast text once per emit; no double announcement.
- **Contrast:** all text uses `--ink` on `--card` (WCAG-AA M0 pair). Error-toast border uses `--cat-9` (M0 category palette, audited WCAG-AA on `--card`).
- **430 px viewport:** FirstBrickCard is full-width up to 430 px; YearHeatmapPreview's MonthCell grid is `transform: scale(0.6)` from M9e's natural size — fits comfortably. Toaster is `max-width: 360px` and centered.
- **No new focus target on the overlay surfaces** (BrandMarkLongPress is the one new interactive element; the brand-mark wrapper button is keyboard-focusable but `onLongPress` is touch-driven — keyboard activation is a no-op, consistent with the brand mark being decorative chrome). VERIFIER may pick a keyboard alternative (Enter-key-hold) but the spec is explicit: "long-press is the only entry."
- **axe:** zero violations expected. VERIFIER runs axe on each new surface at first paint.

### Performance

- **`<FirstBrickCard>` cost:** mount → 3 s → unmount once per user lifetime. Framer spring on a single `motion.div`. Negligible.
- **`<BrandMarkLongPress>` cost:** O(1) `useLongPress` hook + state flip on long-press. No re-render churn (overlay mounts inside the wrapper; the wrapper re-renders only when `overlayVisible` flips).
- **`<YearHeatmapPreview>` cost:** 12 `<MonthCell>` instances; `monthScore` is pure O(days-in-month). Total ~30 ms cold render on a typical handset. Mounts only during long-press (rare).
- **`<Toaster>` cost:** module-level emitter is O(1) emit; single subscriber; one Framer spring per toast. Negligible.
- **Bundle delta:** new files = FirstBrickCard (~50 LOC) + BrandMarkLongPress (~60 LOC) + YearHeatmapPreview (~60 LOC) + Toaster (~80 LOC). Net build size delta < 4 KB minified.
- **Lighthouse Perf:** target ≥ 90 (AC #14). Sandbox-deferred; verified on Vercel preview at Gate #2.

### Commit strategy

- **An undo for delete-block / delete-brick.** The toast announces the action; no in-toast action button.
- **An error toast on the M6 REORDER_BLOCK overlap-rejection path.** PLANNER baseline does NOT wire this; flagged as SG-m7e-05 for VERIFIER. If approved, a one-line addition at the snap-back call site in `<DraggableTimelineBlock>`.
- **An `appliesOn`-aware first-brick detector.** "First brick ever" is a lifetime fact (any brick in `state.blocks[].bricks` or `state.looseBricks` counts), not a daily fact.
- **A migrator step (v3 → v4).** `firstBrickShown` is optional with a deterministic back-fill — per ADR-044, this is an additive within-version extension. No `SCHEMA_VERSION` bump.
- **A new motion token in `lib/motion.ts`.** Sheet-slide is an inline spring; toast slide is an inline timed transition; PRM cadences are inline in CSS keyframes. Matches M7b/M7c/M7d precedent.
- **A new CSS custom property.** Durations are inline.
- **A new asset.** No new image / SVG / font / audio.
- **A new third-party dependency** (a toast library like `sonner` or `react-hot-toast`). PLANNER baseline ships a ~80-LOC module-level emitter — simpler than a dep, fewer bundle bytes.
- **A new clock or `setInterval`.** Auto-dismiss uses one-shot `setTimeout` (matches M4a/M7d pattern).
- **Coupling to M7d's celebrations.** The first-brick card is independent of bloom/fireworks; the user's first brick may not be inside a block that hits 100 %, so the card may fire without a celebration. AC #1 + AC #8.

### Open questions for VERIFIER

All four SG items are **resolved in-plan exactly per the spec's recommendation** (per the dispatch prompt). The following implementation choices are surfaced for VERIFIER ratification — all are plan-level mechanism decisions, not deviations from the spec:

- **Module-level emitter vs. React Context for `<Toaster>`.** Plan baseline ships a module-level `subscribers: Set<fn>` + `toast(...)` function. Alternative: `<ToastProvider>` + `useToast()` hook. Plan baseline is simpler (no Provider, no consumer wiring); alternative is more idiomatic for SSR-heavy apps. M7e is client-only (`<Toaster />` lives inside the `"use client"` AppShell); module-level emitter is safe. VERIFIER picks.

- **First-brick card visibility: derived from state vs. local boolean.** Plan baseline uses a local `firstCardOpen` boolean in `<BuildingClient>` seeded by a `useEffect` that watches the brick-count transition AND the persisted flag. Alternative: derive visibility purely from `state.firstBrickShown !== true && hasAnyBrick(state)` with no local state — simpler, but creates a race when the reducer flips the flag in the same render pass as the brick add (the predicate would briefly be false on the very render that should show the card). Plan baseline is more robust; VERIFIER picks.

- **`<BrandMarkLongPress>` wraps the brand mark in a `<button>` vs. retains the `<div>` element.** Plan baseline upgrades to `<button>` for ADR-031 + a11y. Alternative: leave the `<div>` and attach pointer handlers directly (matches the M0 visual fidelity; loses keyboard focus). Plan baseline is more a11y-correct; alternative is closer to "secret door" affordance (no focus ring). VERIFIER picks; same touch behavior.

- **`<YearHeatmapPreview>` composes `<MonthCell>` directly vs. extends `<YearView>` with a `size` prop.** Plan baseline composes `<MonthCell>` directly to avoid touching M9e's shipped surface. Alternative: extend `<YearView>` with `size="full" \| "preview"` and pass `"preview"` here. Plan baseline has a smaller blast radius (zero M9e regression risk); alternative is closer to a "single source of truth" for the year heatmap. VERIFIER picks.

- **PLANNER-flagged gap: SG-m7e-05 — error toast on M6 REORDER_BLOCK overlap-rejection.** Spec AC #11: "a rolled-back action emits an error toast." M7e's four happy-path dispatch sites are unconditional toast emits. The only "rolled-back" path the spec explicitly references is the ADR-018 quota-exceeded path — which is theoretical today (M8 silently swallows quota errors in `saveState`). M6's `REORDER_BLOCK` overlap-rejection is the closest live "rollback" — the reducer returns `state` unchanged and the UI snap-back animation is the user-facing signal. M7e baseline does NOT wire an error toast at the snap-back call site (no spec'd toast event covers reorder; expanding scope would touch M6 surfaces). VERIFIER: ratify the baseline OR direct PLANNER to wire `toast("Couldn't move block — overlap", "error")` at `<DraggableTimelineBlock>`'s snap-back call site. One-line addition either way.
