## Milestone 7c ‚Äî Hero % count-up on first load ‚Äî Plan

### Context

The hero's `0%` ‚Üí real-% jump happens silently at hydration today (HeroRing's existing two-pass `mounted` flag renders `pct=0` on SSR, then snaps to the real `dayPct(state)` value on the post-mount `setState`). M7c animates that snap: a **1.6 s easeOut tween from `0` to `dayPct(state)`'s integer percent**, fired exactly once on first hydrated paint. Both the SVG `stroke-dashoffset` and the visible numeral ride the same tween so they stay in sync. Subsequent updates (a brick logged, a block deleted) continue to flow through HeroRing's existing per-update 600 ms `stroke-dashoffset` transition ‚Äî M7c does NOT re-fire the count-up. Render-layer only: no schema bump, no new action, no new persisted field. M7c reuses M7a's `useFirstPaintAfterHydration` hook (NOT a new ref) ‚Äî exactly the "ref-machine pattern" the spec calls out ‚Äî so the count-up fires on the same render the stagger fires (concurrent, not serialized ‚Äî SG-m7c-03).

### Feature grouping

m7c is **one feature group** (slug `m7c`), one BUILDER dispatch (ADR-013, ADR-022). `tests.md` IDs use the `m7c` slug (`U-m7c-*`, `C-m7c-*`, `E-m7c-*`, `A-m7c-*`). See ¬ß Test-ID prefix scheme.

### `<HeroRing>` count-up wiring (resolves spec AC #1 stroke ‚Üî numeral sync)

The current HeroRing computes `displayPct = mounted ? pct : 0` and uses it for `dashoffset` math; the numeral is rendered by the **parent (Hero.tsx)** at 72 px and by HeroRing as a fallback when `children` is undefined. Today's split-numeral architecture means a count-up driving HeroRing alone would desynchronize the 72-px parent numeral from the stroke. **The plan resolves this by making HeroRing the single source of truth for the count-up tween value, and exposing it via a render prop OR by rendering the numeral itself.**

**Plan baseline: "HeroRing as the tween owner; numeral renders via children-as-function":**

```tsx
// components/HeroRing.tsx (sketch)
interface Props {
  pct: number;
  firstPaintCountUp?: boolean;
  children?: ReactNode | ((displayPct: number) => ReactNode);
}

export function HeroRing({ pct, firstPaintCountUp = false, children }: Props) {
  const prefersReducedMotion = useReducedMotion();
  const [mounted, setMounted] = useState(false);
  const [displayPct, setDisplayPct] = useState(0);

  useEffect(() => {
    // eslint-disable-next-line react-hooks/set-state-in-effect -- canonical SSR hydration two-pass per SG-m3-17 + M7c count-up
    setMounted(true);
    if (firstPaintCountUp && pct > 0 && !prefersReducedMotion) {
      // Framer Motion's `animate(from, to, options)` returns a controls handle.
      // It writes to a target on every frame; we plug a small subscriber.
      const controls = animate(0, pct, {
        duration: motionTokens.countUp.durationMs / 1000, // 1.6 s
        ease: [0, 0, 0.2, 1], // SG-m7c-02 ‚Äî Framer's "easeOut" cubic-bezier
        onUpdate: (v) => setDisplayPct(v),
      });
      return () => controls.stop();
    }
    // No tween path (PRM, pct===0, or firstPaintCountUp===false):
    setDisplayPct(pct); // snap to final on mount
  }, [firstPaintCountUp, pct, prefersReducedMotion]);

  const dashoffset = (1 - displayPct / 100) * CIRCUMFERENCE;
  const roundedDisplayPct = Math.round(displayPct); // SG-m7c-01 ‚Üí Math.round
  // ‚Ä¶ render ‚Ä¶
  // Children-as-function lets Hero.tsx render the 72-px numeral with the live tween value:
  return (
    <div style={{‚Ä¶}}>
      <svg ‚Ä¶>{/* stroke-dashoffset uses dashoffset, computed from displayPct */}</svg>
      {typeof children === "function" ? children(roundedDisplayPct) : children}
    </div>
  );
}
```

`Hero.tsx` then becomes:

```tsx
<HeroRing pct={pct} firstPaintCountUp={firstPaintCountUp}>
  {(rounded) => (
    <span
      aria-hidden="true"
      className="font-serif-italic text-[72px] leading-[0.85]"
      style={{ fontFamily: "var(--font-display)" }}
    >
      {rounded}%
    </span>
  )}
</HeroRing>
```

Net effect: a **single `displayPct` state** drives BOTH the SVG stroke-dashoffset AND the parent's 72-px numeral via children-as-function. The two CANNOT desync ‚Äî they consume the same React state on the same render (AC #1). On every animation frame, `displayPct` updates ‚Üí React schedules one re-render ‚Üí both the `dashoffset` math and the parent numeral re-render with the same value. The integer-step requirement (AC #2) is satisfied by `Math.round(displayPct)` in the numeral render (the stroke-dashoffset remains floating-point for smooth pixel-perfect arc growth; only the numeral display rounds ‚Äî matches the spec's "numeral updates every animation frame in integer steps" wording: the **numeral** is integer-stepped; the stroke is not).

**Alternative considered + rejected: useMotionValue + motion.text.** Framer Motion's `useMotionValue` + `<motion.span>` could animate without React re-renders. Rejected because (a) the SVG `strokeDashoffset` attribute is not a Framer-driven attribute by default (would require `motion.circle` + `motionValue.on("change")`) ‚Äî equivalent complexity; (b) we'd still need to round the numeral, which means a subscriber anyway; (c) children-as-function is cleaner and trivially testable (the test passes a synchronous `pct` and asserts the rendered numeral progresses through integer values).

**Children backwards-compat:** existing callers pass a `ReactNode` child (the current Hero.tsx 72-px span). HeroRing accepts BOTH `ReactNode` and `(displayPct: number) => ReactNode` via a type union. When a caller passes a plain ReactNode (today's pattern), HeroRing renders the child verbatim ‚Äî byte-identical to today's behavior (the child does not see the tween; but neither does it today). M7c migrates Hero.tsx to children-as-function; every other test using `<HeroRing pct={‚Ä¶} />` (no children) or `<HeroRing pct={‚Ä¶}><‚Ä¶/></HeroRing>` (static ReactNode) continues to work. **Zero existing-test amendment required to HeroRing's `children`-only callers.**

### Components

**`<HeroRing>` (MODIFY ‚Äî `firstPaintCountUp?: boolean` + count-up tween + children-as-function)**

- **Props:** `{ pct: number; firstPaintCountUp?: boolean; children?: ReactNode | ((displayPct: number) => ReactNode); }`. Existing usage with `children` as a plain ReactNode is preserved verbatim.
- **State:** `mounted` (existing two-pass flag) + `displayPct` (new ‚Äî drives both stroke + numeral).
- **Effect:** on mount, if `firstPaintCountUp === true && pct > 0 && !prefersReducedMotion`, kick off `animate(0, pct, ‚Ä¶)` from `motion/react` with `duration: 1.6 s` and `ease: [0, 0, 0.2, 1]` (the Framer canonical easeOut cubic-bezier per SG-m7c-02); each `onUpdate` writes `displayPct` via `setDisplayPct`. On unmount or `pct` change before completion, `controls.stop()` cancels ‚Äî Framer Motion's last-write-wins handles a brick-log-mid-tween cleanly (spec edge case "Brick logged during the count-up"). Otherwise (PRM, pct===0, firstPaintCountUp===false), `setDisplayPct(pct)` snaps to final on mount.
- **Fire-once contract (AC #3).** The effect's dependency array `[firstPaintCountUp, pct, prefersReducedMotion]` means: once `firstPaintCountUp` flips false ‚Üí true on first hydrated paint (driven by `useFirstPaintAfterHydration`), the tween fires. **`useFirstPaintAfterHydration` then returns `false` on every subsequent render** (per its tri-state ref machine ‚Äî `pending ‚Üí staggered ‚Üí done`), so the prop in the BuildingClient call site reverts to `false` for any later re-render. **The effect runs only once because once `firstPaintCountUp` flips back to `false`, the tween branch is not re-entered.** A `pct` change while `firstPaintCountUp === false` triggers the effect's `setDisplayPct(pct)` snap arm ‚Äî which is exactly the existing snap-to-pct behavior (no count-up). To prevent a redundant `setDisplayPct(pct)` snap when the count-up has already landed at `pct` (the count-up's `onUpdate(pct)` final frame), the effect's snap arm is a no-op when `displayPct === pct` (guarded inside the effect body, not in the dependency array). VERIFIER may ratify simpler guard variants ‚Äî the contract is: tween fires once per mount; no re-count-up.
- **Reduced motion (AC #5).** When `prefersReducedMotion === true`, the tween branch is bypassed; `setDisplayPct(pct)` runs on mount and the numeral + stroke render at final value on the very next paint. No per-frame animation. VERIFIER ratifies via `motion/react`'s `useReducedMotion()` hook return value (HeroRing already consumes this ‚Äî see line 25).
- **Existing 600 ms `stroke-dashoffset` transition (per-update path).** Preserved verbatim on the SVG style attribute. When the count-up is NOT running (i.e., `firstPaintCountUp === false` or PRM-active), a `pct` change still smoothly animates the arc via the existing `transition: "stroke-dashoffset 600ms ease-in-out"`. M7c's count-up COEXISTS with the per-update transition ‚Äî they never overlap because (a) the count-up only fires on first paint (one-shot), and (b) once the count-up completes, the per-update transition takes over for every subsequent change.
- **`children` API extension.** Type-union `ReactNode | ((displayPct: number) => ReactNode)`. Implementation: `typeof children === "function" ? children(roundedDisplayPct) : children`. Backwards-compat: every existing test/consumer passing a static node sees no change.

**`<Hero>` (MODIFY ‚Äî passes `firstPaintCountUp` and switches to children-as-function)**

- **Props:** existing `{ dateLabel, dayNumber, totalDays, pct }` PLUS `firstPaintCountUp?: boolean` (default `false`). No state, no effect.
- **Render change.** The 72-px numeral child of `<HeroRing>` becomes a function: `{(rounded) => <span ‚Ä¶ aria-hidden="true" className="font-serif-italic text-[72px] leading-[0.85]" style={{ fontFamily: "var(--font-display)" }}>{rounded}%</span>}`. Default `firstPaintCountUp = false` means existing Hero tests (which omit the prop) render with no count-up ‚Äî byte-identical perceived output to today (the numeral reads `{Math.round(pct)}%` either way; today's hard-coded `Math.round(pct)` becomes `Math.round(displayPct)` where `displayPct === pct` on every render past mount).

**`<BuildingClient>` (MODIFY ‚Äî threads `firstPaintCountUp={stagger}` into Hero)**

- One-line change: `<Hero ‚Ä¶ pct={heroPct} firstPaintCountUp={stagger} />`. The `stagger` variable at line 177 is **the existing source-of-truth**; M7c reuses it (per the dispatch prompt mandate). No new state, no new ref, no parallel ref-machine. VERIFIER: confirm the diff is exactly one prop added to the existing `<Hero />` call site.

### Hero/HeroRing numeral wiring (resolves the split-numeral concern)

The current architecture has Hero.tsx rendering a 72-px numeral as a child of HeroRing, while HeroRing has a fallback 14-px numeral when `children === undefined`. The fallback is **unreached in production** (Hero always passes a child) but exists for HeroRing's standalone unit tests (e.g., `C-m3-006` renders `<HeroRing pct={‚Ä¶} />` directly with no children, exercising the fallback). M7c's plan handles both:

- **Hero call site:** migrates from a static ReactNode child to a render-prop function child. The function receives `displayPct` (the live tween value, integer-rounded) and renders the same 72-px serif span.
- **HeroRing standalone tests:** continue to pass no child. The internal fallback numeral renders `{Math.round(displayPct)}%` ‚Äî which is now the live tween value (was hard-coded `displayPct = mounted ? pct : 0` before). For tests that pass `firstPaintCountUp={false}` (or omit it ‚Äî default `false`), the value resolves to `pct` on the post-mount render, identical to today. VERIFIER: existing C-m3-006 / C-m3-007 cases should pass byte-identical because their render-tree DOM at `firstPaintCountUp` default `false` produces the same numeral output.

The split-numeral consideration is otherwise inert ‚Äî HeroRing is the sole owner of the tween's `displayPct` state, regardless of whether the parent or the fallback renders the visible glyph.

### Resolutions to the 3 Open Spec Gaps

- **SG-m7c-01 ‚Äî RESOLVED ‚Üí `Math.round(displayPct)` at every render site.** Per the spec's recommendation verbatim. Applied at: (a) Hero's render-prop child for the 72-px numeral; (b) HeroRing's fallback 14-px numeral; (c) HeroRing's `aria-label={`Day score: ${roundedPct}%`}` (already uses `Math.round(pct)` today ‚Äî keep that, but switch the source to `displayPct` so the aria-label progresses with the tween). The stroke-dashoffset math at HeroRing's circle element stays floating-point (the stroke is smooth-arc, the numeral is integer-stepped ‚Äî distinct rounding contracts on the same source state). VERIFIER: confirm `Math.round` (not `Math.floor`) is the function in use, and that it is applied consistently across all numeral render sites + the aria-label.

- **SG-m7c-02 ‚Äî RESOLVED ‚Üí Framer Motion's canonical `easeOut` (`cubic-bezier(0, 0, 0.2, 1)`).** Per the spec recommendation verbatim. Implemented as `ease: [0, 0, 0.2, 1]` at the `animate(0, pct, { ease, ‚Ä¶ })` call site inside HeroRing's effect. The `motionTokens.countUp.easing` field stores the string `"easeOut"` (matches the existing `Easing` type union, which already includes `"easeOut"`); the explicit cubic-bezier is materialized at the consumer when calling Framer's `animate`. VERIFIER: confirm Framer's `"easeOut"` alias matches the explicit bezier (they do ‚Äî `[0, 0, 0.2, 1]` is Framer's documented `easeOut` curve).

- **SG-m7c-03 ‚Äî RESOLVED ‚Üí run count-up CONCURRENT with M7a stagger; no serialization.** Per the spec's recommendation verbatim. The Hero is not part of M7a's three-surface stagger (it's its own region, rendered ABOVE the M7a skeleton/real branch in BuildingClient ‚Äî line 485 vs. line 494). Both fire on the same first hydrated paint, driven by the same `useFirstPaintAfterHydration(hydrated)` value ‚Äî they animate different DOM trees in parallel. **No `setTimeout`, no `await`, no chaining, no orchestrator.** Perceived as "everything coming alive at once". VERIFIER: confirm no `setTimeout` / `setInterval` / `Promise` / `MotionValue.onComplete` chain is introduced to serialize them.

### Accessibility

- **`aria-label` continues to read the canonical value.** Today's `aria-label={`Day score: ${Math.round(pct)}%`}` becomes `aria-label={`Day score: ${Math.round(displayPct)}%`}` ‚Äî the aria-label progresses with the tween. This is intentional: AT users hear the score climbing if they happen to be focused on the HeroRing during the count-up. `aria-live="polite"` on the SVG (existing) batches the announcements politely ‚Äî the screen reader speaks the most recent value once the tween settles, not every frame (browsers throttle polite-region updates).
- **Reduced motion (AC #5).** Under `prefers-reduced-motion: reduce`, the tween never starts; the numeral + stroke render at final value on mount. The aria-label reads the final percentage immediately. No motion-only affordance ‚Äî the data appears identically with or without the count-up.
- **No new focus target.** HeroRing remains non-interactive; the numeral is `aria-hidden="true"`. No tab order change.
- **No new `aria-live` region.** The existing `aria-live="polite"` on the SVG is unchanged.
- **Contrast.** Unchanged ‚Äî text on background, accent on background; all M0 token values.
- **430 px viewport.** The 72-px numeral fits horizontally at 430 px (already verified in M3). The count-up does not change the numeral's max width (a 3-digit "100%" is the worst case; verified by today's M3 build).
- **axe.** Zero violations expected. VERIFIER runs axe on the preview at first paint and at tween-completion.

### Performance

- **Per-frame cost.** Framer Motion's `animate(0, pct, ‚Ä¶)` writes via `onUpdate` ~60 times per second; each update calls `setDisplayPct` which schedules one React render. React 19's automatic batching coalesces multiple sub-frame updates into one render per frame. The SVG `strokeDashoffset` attribute update is a compositor-only paint (no reflow); the numeral textNode mutation is a microtask. **Sub-millisecond per-frame budget.** No main-thread spike. 60 fps maintained on a Pixel 7 / Moto G4 baseline (status quo M6 perf budget).
- **Cancellation hygiene.** The effect's cleanup function calls `controls.stop()`; on `pct` change mid-tween or on unmount, the animation halts immediately ‚Äî no memory leak, no orphaned setState on an unmounted component (Framer's `controls.stop()` is synchronous and idempotent).
- **Bundle delta.** `animate` is already in the `motion/react` re-export surface (M7a uses `useReducedMotion`; M2/M3 use `motion.div` + `animate` for sheet transitions). **Zero new asset weight.** The new `motionTokens.countUp` entry is one object ‚Äî negligible bundle size.
- **First Contentful Paint.** Unchanged. The count-up starts AFTER first paint (on the post-mount `useEffect`); FCP is at or below the M6/M7a/M7b baseline.
- **Lighthouse Perf ‚â• 90.** AC #7. Verified on the Vercel preview (deferred E2E).

### Out of scope

### Open questions for VERIFIER

All three SG items are **resolved in-plan exactly per the spec's recommendation** (per the dispatch prompt). The following implementation choices are surfaced for VERIFIER ratification ‚Äî all are plan-level mechanism decisions, not deviations from the spec:

- **PLANNER answer to the dispatch prompt's question ‚Äî "stroke + isolated `useEffect` tween, or both jointly?"** **DECIDED: BOTH JOINTLY, via a single shared `displayPct` state.** The count-up tween writes to one `displayPct` state variable inside HeroRing; the SVG `strokeDashoffset` consumes `displayPct` directly; the visible numeral (parent's 72-px child, rendered via children-as-function) consumes the same `displayPct`. They cannot desync at the React render level ‚Äî they read from the same source-of-truth on every render. This is the cleanest answer to AC #1 ("the numeral and the SVG stroke-dashoffset animate jointly"). Alternative: drive ONLY the stroke from a tween and let the numeral lag ‚Äî explicitly rejected because the spec says "jointly". Alternative: drive each independently (two `animate()` calls) ‚Äî explicitly rejected because they would drift by a frame or two. VERIFIER: ratify the joint-state choice.

- **Children-as-function API extension to HeroRing.** Plan baseline extends `children` to `ReactNode | (displayPct: number) => ReactNode`. Alternative: introduce a separate render-prop `renderNumeral?: (rounded: number) => ReactNode` keeping `children` as plain ReactNode. Plan baseline is fewer props, cleaner API; alternative is more explicit. Both pass the same tests with trivial assertion adjustment. VERIFIER picks.

- **`animate()` from `motion/react` vs. `useMotionValue` + `motion.span`/`motion.circle`.** Plan baseline uses the imperative `animate(0, pct, { onUpdate, ease, duration })` API with a `setState` subscriber. Alternative: declarative `useMotionValue` + `<motion.span style={{ ['--n']: motionValue }}>` + a CSS `attr()` or computed-style binding. The imperative path is simpler to test (Vitest fake timers + `act` + explicit `setState`); the declarative path is closer to Framer idiom but harder to inspect in tests. VERIFIER picks; both pass the same tests with different scaffolding.

- **Easing literal: string alias `"easeOut"` vs. explicit cubic-bezier `[0, 0, 0.2, 1]`.** Plan baseline passes the explicit bezier at the `animate` call site (so the test asserts the literal value), and stores `"easeOut"` in `motionTokens.countUp.easing` (matches the existing `Easing` type union). Alternative: pass the string at the call site too ‚Äî Framer's `"easeOut"` resolves to the same bezier. VERIFIER picks; identical motion.
